# Tidy-Doc

用这个 skill 做"系统化文档整理"，而不是做单点修字。

目标通常是四件事：

- 让入口文档更短、更稳定
- 让规范文档和当前代码现实一致
- 让 skill / guideline / framework / 索引之间的引用闭环
- 清掉会误导后续开发的旧路径、旧术语、旧规则

## 适用任务

以下类型的请求应触发本 skill：

- 精简 `AGENTS.md`、skill、guideline、framework 文档
- 对齐 `docs/Guidelines`、`docs/Framework`、`docs/Development`
- 清理重复规则、失效入口、旧目录名、旧组件落点、旧架构说法
- 修正文档索引、skill 触发范围、文档迁移状态
- 做一轮"文档一致性审计"或"文档收口"

如果只是单个错别字、单条链接、单处措辞修正，不需要完整走本 skill 的全部流程。

## 工作流

### 1. 先盘点入口

优先读取当前任务命中的入口文档：

- `AGENTS.md`
- 相关 `.claude/skills/*/SKILL.md`
- 相关 `docs/Guidelines/*.md`
- 相关 `docs/Framework/*.md`
- 必要时读取 `docs/Development/Active/*.md` 与 `docs/Development/Archived/*.md`

先回答：

- 哪些文档是入口文档
- 哪些文档是当前稳定基线
- 哪些文档只是开发中或历史背景

不要一开始就大面积改所有文档。

### 2. 先找"会误导执行"的问题

优先修这些高价值问题：

- 文档把人带到旧目录、旧文件、旧项目结构
- skill 没覆盖当前真实文档入口
- guideline / framework / 索引说法互相冲突
- 当前代码现实已经变了，但文档还在沿用旧规则
- 一个任务要读很多长文档才能知道当前基线是什么

比起修语气、修行文、修排版，优先处理会误导开发决策的问题。

### 3. 按层收口

默认按这个顺序整理：

1. `AGENTS.md`
2. 对应 skill
3. `docs/Framework` 中的稳定基线文档
4. `docs/Guidelines`
5. `docs/Research`、`docs/Issues`
6. `docs/Development/*`

原因：

- 先收入口，后收正文
- 先收"当前规则"，后收"过程文档"
- 先让检索路径正确，再缩减具体内容

### 4. 精简规则

文档精简时默认保留：

- 文档回答什么问题
- 当前稳定结论
- 必须遵守的边界
- 改动时的最小检查项
- 必要的实现锚点或相关文档链接

优先删除或压缩：

- 大段重复解释
- 与别处重复的规则正文
- 已完成迁移的历史背景长段落
- 未来路线图式展开描述
- 不再触发实际决策的例子

不要把文档压到只剩口号；要保留足够让后续 agent 直接执行的规则。

### 5. 区分三类文档

整理时始终区分：

- `Guidelines`
  - 回答"默认做法 / 门禁 / 施工约束"
- `Framework`
  - 回答"当前稳定基线是什么"
- `Research`
  - 回答"逆向了什么、结论是什么、待确认什么"
- `Issues`
  - 回答"出了什么问题、现场是什么、怎么修的"
- `Development`
  - 回答"当前推进中的主题怎么做"

默认 owner 规则：

- 架构事实、稳定职责、共享协议、框架语义，只保留一个 `Framework` owner
- `Guideline` 不重复定义框架语义，只补默认做法、接入步骤、删旧要求、验证门禁
- `Development` 不承担长期基线；主题结束后默认先迁到 `Archived`

如果一个文档同时承载了三类内容，优先把它收口到单一职责，而不是继续堆。

### 6. 同步索引与 skill

只改正文不够；每次整理后至少同步检查：

- `AGENTS.md` 是否仍准确描述 skill 适用范围
- skill 是否能命中当前应该读的文档
- 当文档变多时，索引和 skill 是否已经按主题分组，而不是继续平铺入口

### 7. 低频残留审计

在主入口收口后，再扫低频残留：

- 旧目录名
- 旧类型名
- 旧架构说法
- 旧日志 / 配置模板
- 文档内部计数或章节引用错误

只改已经会误导开发的残留；纯历史背景可保留，但必须避免被误读成现行规范。

## 默认检查清单

每次文档整理至少检查：

- 是否明确区分入口文档、稳定基线、开发中主题、历史归档
- 是否还有重复发明同一套规则的地方
- 是否还有旧路径、旧目录、旧术语会把人带偏
- 是否还有 skill 覆盖不到的关键文档
- 是否还有索引漏项、失链、错链
- 是否把"当前规则"与"历史背景"混在一起
- 是否在精简后仍保留可执行的最小规则

## 输出要求

完成一次整理后，输出应尽量说明：

- 这次收了哪些入口或基线文档
- 修掉了哪些会误导执行的问题
- 哪些文档只是被精简，哪些文档被重新定位
- 是否还剩下一批低频残留未处理

## 边界

- 不要为了"整洁"重写所有文档语气
- 不要把历史文档全部删掉；先区分是否仍有参考价值
- 不要把 guideline、framework、development 混成一份超级总文档
- 不要制造新的辅助 README、CHANGELOG、整理说明文档
- 如果文档与代码现实不一致，先指出不一致，再更新文档或代码，不要继续沿用旧说法
