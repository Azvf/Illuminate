# Impact Analysis

用这个 skill 在改动代码前做系统性影响评估，确认改动波及范围、识别回归风险、确认改动必要性。

## 触发条件

- 改动涉及跨模块、跨层、跨包的修改
- 改动会修改 public API 或共享契约
- 改动涉及核心路径（启动、主流程、关键业务链路）
- 用户要求评估改动影响
- 改动范围超过单个函数或单个文件

## 默认先读

- `docs/Guidelines/工程约定.md`
- `docs/Framework/`（如涉及架构事实）

## 核心规则

1. **先评估再实现**：改动前必须确认影响范围，不允许边改边发现问题。
2. **分层标注影响**：改动影响必须按模块/层/包显式列出，不用"可能影响"模糊描述。
3. **区分直接与间接影响**：直接调用方是直接影响；依赖直接调用方的模块是间接影响。两者必须分开标注。
4. **回归风险必须可测试**：每个识别的风险点必须有对应的验证方式（已有测试或需新增测试）。

## 工作流

### 1. 确认改动意图

明确回答：

- 改动什么（具体到函数/类/模块）
- 为什么改（root cause 或需求驱动）
- 改动后的预期行为是什么

### 2. 标注直接影响

Grep / 搜索所有直接调用方：

| 调用方 | 调用方式 | 是否需要改动 | 改动内容 |
|--------|---------|------------|---------|

对每个直接调用方，标注：

- 调用方式（直接调用、接口实现、DI 注入、反射）
- 是否需要同步改动
- 如需改动，改动内容是什么

### 3. 追溯间接影响

从直接影响出发，追溯二级、三级调用方，直到没有新的受影响模块：

```
改动点 → 直接调用方A → 二级调用方X
                   → 二级调用方Y → 三级调用方Z
        → 直接调用方B
```

标注每一层是否需要改动。

### 4. 识别回归风险

对每个受影响模块，识别：

- 是否有已有测试覆盖该调用路径
- 改动是否会破坏已有测试
- 改动是否引入新的边界条件
- 改动是否影响并发/线程安全/缓存一致性

| 受影响模块 | 回归风险 | 已有测试 | 需新增测试 |
|-----------|---------|---------|-----------|

### 5. 确认改动范围最小化

回答：

- 改动是否可以缩小范围（只改必要部分）
- 是否存在更简单的替代方案（不改 API、不跨层）
- 改动是否 1:1 对应 root cause 或需求

### 6. 输出影响评估

汇总：

- 直接受影响模块数
- 间接受影响模块数
- 需新增测试数
- 改动范围是否最小化

## 完成前确认

- 是否已列出所有直接调用方
- 是否已追溯间接影响到无新模块
- 每个回归风险是否有对应验证方式
- 改动范围是否已最小化
- 是否已确认改动 1:1 对应 root cause 或需求

## 默认产出

- 直接影响表（调用方、调用方式、改动内容）
- 间接影响链路图
- 回归风险表（模块、风险、已有测试、需新增测试）
- 改动范围最小化确认
- 影响评估结论（是否可安全实施）

## 边界

- 不负责实现代码——本 skill 聚焦影响评估，完成后切回对应实现类 skill
- 不负责 root cause 分析——那是 `backtrack-root-cause` 的职责；本 skill 假设 root cause 已定位或需求已明确
- 不负责性能影响评估——那是 `perf-profile` 的职责；本 skill 聚焦正确性和范围影响
