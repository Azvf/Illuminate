# Backtrack Root Cause

当排障或开发陷入"反复打补丁 — 无效 — 再打补丁"的死循环时，用此 skill 强制回到 root cause 分析起点，完整重梳理后再决定修复。

## 触发短语

用户发出以下信号之一时，立即激活：

- "不要反复做补丁修改"
- "先重新完整梳理 root cause"
- "不要再打补丁了"
- "回到根因分析"
- "你一直在死胡同里绕"

## 核心原则

- **补丁叠加不等于修复**：一个正确修复替换 N 个补丁。
- **Symptom ≠ Trigger ≠ Root Cause**：三者必须分开写清，不得混为一谈。
- **每次只能有一个 root cause**：如果结论中有多个"可能"，说明还没找到。

## 默认先读

- `docs/Guidelines/工程约定.md`

## 流程

### Phase 0：立即停手

- 停止正在进行的任何代码修改
- 丢弃当前尚未提交的所有补丁修改（或 stash 到一边）

### Phase 1：回溯记录（Retrospective）

先写下本轮已尝试过的所有修复尝试，每条包含：

| # | 改了什么（diff 摘要） | 预期效果 | 实际结果 | 为什么失败 |
|---|----------------------|---------|---------|-----------|

如果写不出"为什么失败"，说明对该次尝试的理解不足，先补日志/测试验证再继续。

### Phase 2：返回现象（Return to Symptom）

从最早的原始现象重新出发，回答三个问题：

1. **可观察到的第一个异常是什么？**（不是推断，是事实）
2. **复现条件是什么？**（最小操作序列或输入）
3. **正常行为应该是什么？**（同一输入下的预期输出）

这些回答中不得包含任何"可能""应该是""推测"等假设性措辞。如果某一项只能用假设回答，先取得事实再继续。

### Phase 3：调用链梳理

从入口到失败点画出完整调用链：

```
[入口] → [环节A] → [环节B] → ... → [偏离点] → [异常表现]
```

每一环节必须标注：
- 输入与输出（能数值化就数值化）
- 是否经过日志/调试验证（已验证 / 仅代码推断）
- 如果有两个以上环节仅靠代码推断，先补日志或测试脚本，不得继续向下推进

### Phase 4：三段式 Root Cause

- **Symptom（症状）**：用户/测试能直接观测到的异常表现
- **Trigger（触发器）**：本次复现中使根因暴露的具体条件
- **Root Cause（根因）**：症结所在的本质缺陷（代码缺陷、设计缺陷、契约缺失）

Root Cause 必须满足：
- 是可证伪的命题（能设计一个测试验证它的存在或不存在）
- 不是现象的换个说法
- 解释了所有已观测到的失败，而不是只解释了其中一部分

### Phase 5：修复设计

- 修复必须 **1:1 对应 Root Cause**
- 如果修复代码影响范围超过 root cause 定位的模块/函数，必须先说明为什么需要扩大范围
- 写出修复前后的行为对比（旧行为 → 新行为）
- 设计一个最小验证方法，确认修复消除了 root cause

### Phase 6：实施与验证

- **只做一次修复**，不再叠加补丁
- 先跑最小验证（Phase 5 设计的验证方法）
- 再跑相关测试分组
- 最后跑全量（如适用）
- 修复不通过 → 回到 Phase 3，不得回到 Phase 6 继续改代码

## 完成前至少确认

- 是否已完整写出 Phase 1 回溯记录（所有已尝试的失败修复）
- 是否已用可观测事实重新描述原始现象（不含推断）
- 是否已画出完整调用链且每个环节有日志/测试支撑
- 是否已区分 Symptom / Trigger / Root Cause
- Root Cause 是否可证伪、解释全部失败、不是现象的换个说法
- 修复是否 1:1 对应 Root Cause
- 修复后是否不再依赖运气、顺序、延迟或放宽断言

## 边界

- 不负责补日志或写测试脚本本身——那是 `layer-debug` 或对应领域 skill 的职责；本 skill 负责的是"回到 root cause 分析起点"的决策和框架
- 不负责原始问题的具体技术领域——本 skill 是元流程 skill，完成后切回对应领域的 skill 继续
