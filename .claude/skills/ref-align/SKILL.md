# Ref-Align

用这个 skill 做底层功能模块开发，核心原则是**禁止启发式，必须逆向对齐**。

## 适用场景

- 对接第三方 SDK、平台 API、服务端接口
- 需要对齐竞品或参照产品的行为策略
- 解析外部数据格式、协议、文件结构
- 实现与外部系统交互的核心逻辑

## 默认先读

- `docs/Guidelines/工程约定.md`

## 核心规则：禁止启发式，必须逆向对齐

这是本 skill 最重要的规则，优先于所有其他规则：

**不允许使用 heuristic（启发式）算法来实现核心功能逻辑。**

当有产品参照（如竞品 app、参照 SDK、第三方 API、协议规范）时：

1. **先逆向**：必须先逆向分析参照产品的对应功能，搞清楚它的实际策略、数据结构、匹配规则、判断条件
2. **再对齐**：在核心逻辑上与参照产品的功能策略对齐，而不是自己发明启发式方案
3. **明确标注**：如果某个环节无法逆向确认，必须显式标注为"待逆向确认"，不能默认用猜测填补

具体禁止的做法：

- 用模糊匹配、相似度阈值、概率推断来替代确定性的解析/查找逻辑
- 用"看起来大概对"的近似算法来替代逆向确认过的精确规则
- 在没有逆向证据的情况下，猜测参照产品的行为并据此实现
- 在逆向结论和自研方案之间，默认选自研方案

允许使用启发式的唯一场景：

- 参照产品本身不存在对应功能（全新能力），且已明确确认没有可参照的策略
- 即使此时，也必须在代码注释和提交说明中标注"无参照，自研策略"

## 清除冗余与双轨代码

开发过程中发现以下代码，必须在本次改动中一并清除，不留到下次：

- **废弃代码**：已被新实现替代但未删除的旧路径、旧方法、旧分支
- **Heuristic 残留**：之前用启发式实现、现已有逆向对齐的精确实现后，旧的 heuristic 代码必须删除
- **双轨并行**：同一功能同时存在新旧两套实现，确认新路径已稳定后，必须删除旧路径，不允许长期双轨并存
- **确认无用的辅助代码**：仅服务于已删除功能的 helper、extension、DTO、test fixture

判断标准：如果删除后不影响当前功能和测试，就应该删除。有疑问时先标注 `// TODO: confirm removal`，但不能无限期搁置。

## 默认开发顺序

1. 先判断当前任务属于哪个模块，确认分层位置
2. 如果有产品/API 参照，先完成逆向分析再写实现
3. 先定义契约（接口 / typed request-response），再写实现
4. 先补最小可验证的单元测试骨架，再填充实现
5. 实现时优先复用模块内已有的基础设施
6. 涉及 I/O 密集或大规模数据处理时，先确认是否需要流式/批量处理，不默认全量加载

## 当任务涉及产品参照时

1. 先确认参照产品的具体功能入口和行为链路
2. 定位参照产品对应的实现（代码、数据文件、协议格式等）
3. 提取核心策略：判断条件、匹配规则、数据结构、处理顺序
4. 记录逆向结论到 `docs/Research/`（如果是新的研究主题）或开发文档中
5. 基于逆向结论实现功能，确保核心逻辑对齐
6. 实现完成后用参照产品的已知输出做交叉验证

## 每次至少确认

- 当前功能是否有产品/API 参照，参照策略是否已逆向确认
- 是否有未经逆向确认就使用的 heuristic 逻辑
- 代码放置是否符合分层约束
- 是否引入反向依赖或越层引用
- 新增的 public API 是否最小必要
- 是否需要补 `docs/Research` 逆向文档或 `docs/Development/Active` 开发文档
- 测试是否覆盖核心解析/匹配/查找路径
- 改动范围内是否存在应清除的废弃代码、heuristic 残留或双轨并行路径

## 默认产出

- 功能实现代码
- 逆向对齐说明（如有产品/API 参照）
- 契约变更说明（如涉及跨模块 API）
- 单元测试
- 必要的文档更新
