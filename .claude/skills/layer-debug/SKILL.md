# Layer Debug

用这个 skill 做功能链路的端到端验证排障。

**以日志为核心依据**：先收集并与问题对齐的运行时日志（阶段、路径、关键字段与异常等），必要时提高日志粒度后再读；在此基础上写可执行测试脚本直接调用底层接口，用真实数据跑出可与日志逐项对照的结果。静态阅读代码只用于提出假设，不得在未与日志、脚本结论对齐的情况下认定为 root cause。

## 触发条件

- 数据遗漏、字段缺失、结果不完整
- 解析错误、格式不匹配、编码问题
- 缓存/索引与预期不符
- 调用链路断裂（某一环节未执行、执行顺序错误）
- 修复后需要用日志对照 + 可执行脚本做端到端确认

## 核心规则

**排障必须以运行时日志为核心依据**（时间序、阶段、关键字段须与现象对齐）。在能取得真实数据的前提下，须再产出可执行的测试脚本（或等价的可查数据断言），使结论可与日志逐项对照。

**禁止**在未对齐日志现场的情况下，仅凭静态代码阅读宣称功能已修复或根因已定。

可执行验证须满足：

1. 脚本必须调用真实接口（不使用 mock）
2. 脚本必须使用真实数据作为输入（不使用合成数据）
3. 脚本必须输出 PASS/FAIL 判定和关键数值，不输出模糊描述
4. 脚本退出码必须反映测试结果：0 = 全部通过，非 0 = 有失败

## 工作流

### 阶段 1：复现问题

1. 确认问题的具体表现（缺了什么、哪里不对）
2. **提取并对齐相关运行时日志**（若不足则先补日志再复现）
3. 准备真实数据文件（从用户/测试环境复制到隔离路径）
4. 建立 ground truth（从真实数据中提取预期结果）

### 阶段 2：分层定位断裂环节

自底向上逐层验证，找到第一个失败的环节：

| 层级 | 验证内容 | 方法 |
|------|----------|------|
| **L0 原始层** | 原始数据/二进制/网络包中是否存在目标内容 | 在原始数据中搜索已知特征模式 |
| **L1 解析层** | 解析器能否正确提取出结构化数据 | 调用解析 API 检查输出 |
| **L2 存储层** | 缓存/数据库/本地存储中是否有对应 entry | 直接查询存储 |
| **L3 业务逻辑层** | 业务处理是否正确（过滤、转换、组合） | 检查业务模块输出 |
| **L4 集成层** | 跨模块/跨进程/跨网络调用是否完整 | 端到端调用检查 |
| **L5 表现层** | 最终输出/UI 中是否包含目标内容 | 检查最终输出 |

当某一层失败时，断裂点已定位，无需继续往上验证。

### 阶段 3：修复并验证

1. 在修复代码后，重新运行测试脚本
2. 确认之前失败的层级现在通过
3. 确认端到端测试（L5）通过
4. 运行全量单元测试确认无回归

## 测试脚本规范

### 存放位置

```
scripts/test/<issue-name>/
```

### 运行方式

```
# 按平台运行脚本
```

### 代码结构

```
1. 文件头注释：说明验证的功能链路和预期行为
2. 参数解析：从 args 获取真实数据路径
3. Ground truth 构建：从真实文件中提取预期结果
4. 分层测试：每个测试验证链路中的一个环节
5. 端到端测试：调用完整流程确认最终结果
6. Summary：汇总 PASS/FAIL 计数，返回退出码
```

## 关键约束

- **只用 public API**：测试脚本是独立项目，无法访问 `internal` 类；如需验证 internal 逻辑，用特征模式搜索或存储查询替代
- **不修改被测代码来适配测试**：测试脚本应对当前代码做黑盒验证，不能为了测试而改访问级别或加测试专用出口
- **临时目录隔离**：每个测试用例使用独立的临时目录，结束后清理，避免状态泄漏

## 完成前确认

- 是否与当前问题对齐了足够的运行时日志（阶段、字段、序列）
- 测试脚本是否使用真实数据（不是合成数据）
- 测试脚本是否有明确的 PASS/FAIL 判定（不是只打印信息）
- 测试脚本退出码是否正确（0 = pass, 非 0 = fail）
- 端到端测试是否覆盖了完整的功能链路（不是只测中间环节）
- 分层测试是否帮助定位了具体的断裂环节
- 修复后是否重新运行测试脚本确认通过
- 全量单元测试是否无回归

## 默认产出

- `scripts/test/<issue-name>/` 下的可执行测试脚本
- 分层断裂定位报告（哪一层失败、失败的具体数据）
- 修复代码（如需要）
- 修复后测试脚本的重新运行结果

## 边界

- 不负责性能链路排查（启动慢、加载慢、滚动卡顿）——那属于 `perf-profile` 的职责
- 不负责 root cause 分析的流程框架——那属于 `backtrack-root-cause` 的职责；若反复打补丁陷入死循环，切到 `backtrack-root-cause`
- 本 skill 聚焦功能链路的分层验证与可执行脚本排障，完成后切回对应实现类 skill
